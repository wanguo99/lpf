# Kconfig Integration for ES-Middleware

## Overview

This document describes the Buildroot-style Kconfig integration for ES-Middleware. The integration provides:

1. **Automatic kconfig tools building** (conf, mconf, nconf)
2. **Robust configuration loading** with dependency tracking
3. **Custom CMake targets** for configuration management
4. **Auto-generated headers** (autoconf.h) for C code
5. **CMake variable export** and caching

## Architecture

```
ES-Middleware/
├── cmake/
│   ├── Kconfig.cmake          # Main integration module
│   └── KconfigTools.cmake     # Tool building module
├── scripts/kconfig/           # Buildroot kconfig tools source
│   ├── CMakeLists.txt        # Build configuration for tools
│   ├── conf.c                # Console configuration tool
│   ├── mconf.c               # Menuconfig (ncurses GUI)
│   ├── nconf.c               # Nconfig (alternative GUI)
│   └── ...                   # Parser and support files
├── configs/                   # Defconfig files
│   ├── ccm_h200_100p_am625_debug_defconfig
│   ├── tests_x86_full_defconfig
│   └── ...
└── Kconfig                    # Root Kconfig file
```

## Quick Start

### 1. Initial Setup

```bash
# Create build directory
mkdir build && cd build

# Configure with a defconfig
cmake -DCMAKE_BUILD_TYPE=Debug ..

# Or load specific defconfig
cmake -DDEFCONFIG=../configs/tests_x86_full_defconfig ..

# Build
make -j$(nproc)
```

### 2. Interactive Configuration

```bash
# Open menuconfig GUI
make menuconfig

# Regenerate build files with new configuration
cmake .

# Rebuild
make -j$(nproc)
```

### 3. Load Different Configuration

```bash
# Using make target
make defconfig DEFCONFIG=../configs/ccm_h200_100p_am625_release_defconfig
cmake .
make -j$(nproc)

# Or using CMake target
make ccm_h200_100p_am625_release_defconfig
cmake .
make -j$(nproc)
```

## CMake Integration

### Basic Usage

Add to your root `CMakeLists.txt`:

```cmake
cmake_minimum_required(VERSION 3.16)

# Include Kconfig module
include(cmake/Kconfig.cmake)

# Configure Kconfig integration
kconfig_config(
    KCONFIG_FILE "${CMAKE_SOURCE_DIR}/Kconfig"
    CONFIG_FILE "${CMAKE_BINARY_DIR}/config/.config"
    AUTOCONF_H "${CMAKE_BINARY_DIR}/config/autoconf.h"
    DEFCONFIG "${CMAKE_SOURCE_DIR}/configs/default_defconfig"  # Optional
    ENV_VARS "SDK_PATH=${CMAKE_SOURCE_DIR};BUILD_TYPE=Debug"
)

project(MyProject C)

# Use CONFIG_* variables
if(CONFIG_PRODUCT_CCM)
    add_subdirectory(products/ccm)
endif()

if(CONFIG_BUILD_TESTING)
    enable_testing()
    add_subdirectory(tests)
endif()
```

### Function Reference

#### kconfig_config()

Main configuration function. Parameters:

- `KCONFIG_FILE` - Path to root Kconfig file (required)
- `CONFIG_FILE` - Path to .config file (default: `${CMAKE_BINARY_DIR}/.config`)
- `AUTOCONF_H` - Path to autoconf.h output (default: `${CMAKE_BINARY_DIR}/config/autoconf.h`)
- `DEFCONFIG` - Default defconfig to load if no .config exists (optional)
- `ENV_VARS` - List of environment variables (format: "VAR=VALUE;VAR2=VALUE2")

#### kconfig_check_enabled()

Check if a config option is enabled:

```cmake
kconfig_check_enabled(OSAL osal_enabled)
if(osal_enabled)
    message(STATUS "OSAL is enabled")
endif()
```

#### kconfig_print_config()

Print all CONFIG_* variables (for debugging):

```cmake
kconfig_print_config()
```

## Make Targets

The integration creates several useful targets:

### menuconfig

Open interactive configuration GUI:

```bash
make menuconfig
cmake .  # Regenerate build files
make     # Rebuild
```

### defconfig

Load a defconfig file:

```bash
make defconfig DEFCONFIG=configs/tests_x86_full_defconfig
cmake .
make
```

### savedefconfig

Save current configuration as minimal defconfig:

```bash
make savedefconfig DEFCONFIG=configs/mynew_defconfig
```

### oldconfig

Update .config with new symbols (non-interactive):

```bash
make oldconfig
cmake .
make
```

### alldefconfig

Set all options to their defaults:

```bash
make alldefconfig
cmake .
make
```

### listnewconfig

List new configuration symbols since last update:

```bash
make listnewconfig
```

### <name>_defconfig

Each defconfig file in `configs/` gets its own target:

```bash
make ccm_h200_100p_am625_debug_defconfig
make tests_x86_full_defconfig
make tests_x86_minimal_defconfig
```

## Generated Files

The integration generates three output files:

### 1. .config

Kconfig configuration file (Buildroot format):

```kconfig
CONFIG_OSAL=y
CONFIG_HAL=y
CONFIG_ARCH_X86_64=y
CONFIG_BUILD_TESTING=y
# CONFIG_BUILD_SHARED is not set
CONFIG_MAX_THREADS=32
CONFIG_PROJECT_NAME="ES-Middleware"
```

### 2. autoconf.h

C header with CONFIG_* macros:

```c
/* Automatically generated by Kconfig - do not edit */
#ifndef AUTOCONF_H
#define AUTOCONF_H

#define CONFIG_OSAL 1
#define CONFIG_HAL 1
#define CONFIG_ARCH_X86_64 1
#define CONFIG_BUILD_TESTING 1
#define CONFIG_MAX_THREADS 32
#define CONFIG_PROJECT_NAME "ES-Middleware"

#endif /* AUTOCONF_H */
```

**Note**: This header is automatically included in all C files via `-include` flag.

### 3. config.cmake

CMake variables (for CMakeLists.txt):

```cmake
# Automatically generated by Kconfig - do not edit
set(CONFIG_OSAL TRUE CACHE BOOL "" FORCE)
set(CONFIG_HAL TRUE CACHE BOOL "" FORCE)
set(CONFIG_ARCH_X86_64 TRUE CACHE BOOL "" FORCE)
set(CONFIG_BUILD_TESTING TRUE CACHE BOOL "" FORCE)
set(CONFIG_BUILD_SHARED FALSE CACHE BOOL "" FORCE)
set(CONFIG_MAX_THREADS 32 CACHE STRING "" FORCE)
set(CONFIG_PROJECT_NAME "ES-Middleware" CACHE STRING "" FORCE)
```

## Dependency Tracking

The integration automatically tracks dependencies:

- CMake reconfigures when `.config` changes
- `autoconf.h` is regenerated when `.config` changes
- All C files automatically include `autoconf.h`

## Using CONFIG_* in Code

### In C Code

Access configuration via macros:

```c
#include "osal.h"  // autoconf.h is already included

void my_function(void) {
#ifdef CONFIG_OSAL
    osal_init();
#endif

#if CONFIG_MAX_THREADS > 16
    printf("High thread count: %d\n", CONFIG_MAX_THREADS);
#endif

#ifdef CONFIG_PROJECT_NAME
    printf("Project: %s\n", CONFIG_PROJECT_NAME);
#endif
}
```

### In CMakeLists.txt

Use CMake variables:

```cmake
# Boolean options
if(CONFIG_OSAL)
    add_subdirectory(core/osal)
endif()

# Numeric options
if(CONFIG_MAX_THREADS GREATER 16)
    message(STATUS "High thread count configured")
endif()

# String options
message(STATUS "Building ${CONFIG_PROJECT_NAME}")

# Helper function
kconfig_check_enabled(BUILD_TESTING testing_enabled)
if(testing_enabled)
    enable_testing()
endif()
```

## Defconfig Best Practices

### Creating a Defconfig

1. Configure your build:
   ```bash
   make menuconfig
   ```

2. Save as minimal defconfig:
   ```bash
   make savedefconfig DEFCONFIG=configs/my_product_defconfig
   ```

3. Commit the defconfig:
   ```bash
   git add configs/my_product_defconfig
   git commit -m "Add my_product defconfig"
   ```

### Naming Convention

Follow the pattern: `<product>_<platform>_<variant>_defconfig`

Examples:
- `ccm_h200_100p_am625_debug_defconfig` - CCM product, H200 platform, AM625, debug build
- `ccm_h200_100p_am625_release_defconfig` - Same but release build
- `tests_x86_full_defconfig` - Test suite, x86, all features enabled
- `tests_x86_minimal_defconfig` - Test suite, x86, minimal configuration

### Updating Defconfigs

When Kconfig changes (new options added):

1. Load existing defconfig:
   ```bash
   make ccm_h200_100p_am625_debug_defconfig
   ```

2. Update with new symbols (uses defaults):
   ```bash
   make oldconfig
   ```

3. Save updated defconfig:
   ```bash
   make savedefconfig DEFCONFIG=configs/ccm_h200_100p_am625_debug_defconfig
   ```

4. Commit:
   ```bash
   git add configs/ccm_h200_100p_am625_debug_defconfig
   git commit -m "Update defconfig with new symbols"
   ```

## Troubleshooting

### Tools Not Built

**Symptom**: `KCONFIG_CONF_EXECUTABLE` is empty

**Solution**: Install ncurses development headers:
```bash
sudo apt install libncurses-dev flex bison
```

Then reconfigure:
```bash
rm -rf build
mkdir build && cd build
cmake ..
```

### menuconfig Doesn't Open

**Symptom**: `make menuconfig` fails or target doesn't exist

**Cause**: mconf failed to build (symbol conflicts with ncurses)

**Solution**: Use `nconf` instead:
```bash
make nconfig  # Alternative ncurses interface
```

Or use the Python-based menuconfig:
```bash
python3 build.py menuconfig
```

### Configuration Not Applied

**Symptom**: Changes in menuconfig don't affect build

**Cause**: Forgot to regenerate build files

**Solution**: Always run `cmake .` after configuration changes:
```bash
make menuconfig
cmake .  # Regenerate
make     # Rebuild
```

### CONFIG_* Variables Not Defined

**Symptom**: CMake error about undefined CONFIG_XXX

**Cause**: No .config loaded

**Solution**: Load a defconfig first:
```bash
make defconfig DEFCONFIG=configs/tests_x86_minimal_defconfig
cmake .
```

### autoconf.h Not Found

**Symptom**: Compilation fails, can't find autoconf.h

**Cause**: File not generated or wrong path

**Check**: Verify file was generated:
```bash
ls -l build/config/autoconf.h
```

**Solution**: Ensure kconfig_config() is called before project():
```cmake
include(cmake/Kconfig.cmake)
kconfig_config(...)  # Before project()
project(MyProject)
```

## Integration with Existing Build Systems

### With Python build.py

The existing `build.py` script can coexist with CMake integration:

```bash
# Python-based workflow (unchanged)
python3 build.py config ccm_h200_100p_am625_debug_defconfig
python3 build.py build

# Or CMake-based workflow (new)
cd build
make ccm_h200_100p_am625_debug_defconfig
cmake .
make -j$(nproc)
```

Both workflows use the same `.config` file and generate compatible outputs.

### Migration Path

1. **Phase 1**: Both systems coexist (current)
   - `build.py` uses Python genconfig.py
   - CMake uses Buildroot kconfig tools

2. **Phase 2**: Gradual transition
   - New developers use CMake workflow
   - Existing scripts continue to work

3. **Phase 3**: Full CMake (future)
   - Deprecate `build.py` for configuration
   - Keep for backward compatibility

## Advanced Usage

### Custom Kconfig Targets

Add custom targets for common workflows:

```cmake
# Quick test configuration and build
add_custom_target(quick-test
    COMMAND ${CMAKE_COMMAND} -E cmake_echo_color --cyan "Loading test configuration..."
    COMMAND ${KCONFIG_CONF_EXECUTABLE} --defconfig=${CMAKE_SOURCE_DIR}/configs/tests_x86_minimal_defconfig ${KCONFIG_FILE}
    COMMAND ${CMAKE_COMMAND} .
    COMMAND ${CMAKE_COMMAND} --build . --parallel
    WORKING_DIRECTORY ${CMAKE_BINARY_DIR}
    COMMENT "Quick test build..."
)
```

### Conditional Features

Use Kconfig for fine-grained feature control:

```cmake
if(CONFIG_FEATURE_LOGGING)
    target_compile_definitions(myapp PRIVATE ENABLE_LOGGING)
endif()

if(CONFIG_OPTIMIZATION_LEVEL EQUAL 3)
    target_compile_options(myapp PRIVATE -O3 -march=native)
endif()

if(CONFIG_ENABLE_PROFILING)
    target_compile_options(myapp PRIVATE -pg)
    target_link_options(myapp PRIVATE -pg)
endif()
```

### Multi-Configuration Builds

Build multiple configurations in parallel:

```bash
# Configuration 1: Debug
mkdir build-debug && cd build-debug
cmake -DDEFCONFIG=../configs/ccm_debug_defconfig ..
make -j$(nproc)
cd ..

# Configuration 2: Release
mkdir build-release && cd build-release
cmake -DDEFCONFIG=../configs/ccm_release_defconfig ..
make -j$(nproc)
cd ..
```
